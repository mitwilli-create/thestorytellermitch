The job search, run as production software.career-ops

A job-search automation system that scans ATS portals, evaluates roles against a real CV corpus, generates tailored materials, and tracks everything: a production fork of the open-source santifer/career-ops, extended with a multi-model agent fleet of roughly 50 scheduled jobs on macOS launchd. It has powered 740+ role evaluations. This is not a portfolio demo; it is the system I run my own search on, every day, which is also why the repo stays private. The operating standard is imported from live news and Google comms: daily deadlines, nothing silently wrong.

The spreadsheet job search
Postings found by scrolling, evaluated by mood, tracked in a sheet that goes stale in a week. Every application starts from scratch, and nothing ever tells you your targeting is off.
career-ops
Portals scanned on schedule at zero token cost. Every role scored against the same CV corpus with the same rubric. Dead postings swept automatically, duplicates merged by invariant, materials generated per role, and a daily digest that says what actually needs attention.
What it does

A fork, extended into a fleet. The foundation is santifer/career-ops, an open-source job-search pipeline. The production fork extends it with the operating layer a daily-driver system needs: roughly 50 launchd agents covering portal scans, posting-liveness sweeps, data-integrity checks, nightly enrichment, health monitoring, and a morning email digest. The fork stays current with upstream while the personal data underneath stays gitignored and local.

Evaluation with a rubric, not a vibe. Each posting is triaged cheaply first, then batch-evaluated by stronger models against the real CV corpus, producing a scored report with an explicit recommendation. Hard-disqualifier and compensation-floor gates run as deterministic code after the model, because a scoring rule that matters should not depend on a model's mood. Low-fit roles get told "don't apply."

Operated like production. The fleet has the discipline production systems have: spend caps and kill switches on every autonomous agent, append-only audit logs, regression detection with seeded baselines, a health agent that checks quotas and credentials daily, and a bug ledger with its own resolution pipeline. When something silently breaks, a watcher notices before I do.

Private on purpose. The repo carries my actual pipeline: applications, contacts, evaluation reports, tailored materials. That data is gitignored and never committed, and the fork itself stays private. The architecture is worth showing; the life inside it is not for publishing. I walk the system end to end on request.

How it works
ATS portals (Greenhouse · Ashby · Lever APIs, zero-token scan) │ ▼ [1] triage (cheap models + deterministic disqualifier gates) │ ▼ [2] evaluation (multi-model batch, scored vs the real CV corpus) │ ▼ [3] tracker + reports → tailored materials per role │ ▼ [4] dashboard + daily digest (what needs attention today) ▲ └── ~50 launchd agents (scans · liveness sweeps · health checks · regression guard)
01
Scan without spending
The portal scanner talks to Greenhouse, Ashby, and Lever APIs directly, so discovering new postings costs zero tokens. Dedup history keeps the same posting from being processed twice.
02
Triage cheap, evaluate deep
Fast models do the first pass; strong models do the scored evaluation in batches. Deterministic gates run after the model, so hard rules like disqualifiers and compensation floors are enforced by code, not by hoping the model took them seriously.
03
Generate the materials
For roles worth pursuing, the system builds an apply pack: tailored CV, cover letter, form fields. Generated prose passes through deterministic grounding gates that block fabricated claims and off-limits phrasing before anything is submission-ready.
04
Track everything
One tracker holds every application with canonical statuses, linked reports, and notes. Liveness sweeps discard postings that die; dedupe invariants keep one row per role; nothing depends on memory.
05
Watch the watchers
A regression guard diffs the system against seeded baselines daily, a health agent checks credentials, quotas, and job heartbeats, and a morning digest reports it all. Autonomy is bounded by spend caps and kill switches on every agent.
Why it matters

Anyone can demo an agent. Operating a fleet of them for months, on data you personally depend on, is a different skill: it forces the unglamorous parts, the kill switches, the audit logs, the dedupe invariants, the honest scoring that tells you not to apply. career-ops is where I practice that skill daily. The system's strongest credential is that its owner trusts it with his own job search.

Operated software, not a demo. The proof is not a screenshot; it is that the system has to be right every morning, for me.
