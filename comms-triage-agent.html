Case study · internal-comms triage

Judgment at intake, before a human reads it.comms-triage-agent

Senior engineers generate a steady stream of comms requests that don't need a human first pass, but do need judgment. This agent applies that judgment at intake: three sequential prompts classify, revise, and escalate every request, with conditional knowledge-base loading at each step. A confidence threshold routes anything the model is unsure about to a human instead of letting it guess. This is the sanitized public demo of a system designed for a large engineering organization, and every performance number on this page is a labeled design target, with the math exposed so you can recalculate it for your own org. The triage judgment encoded in these prompts is the judgment I applied by hand as a comms lead, after years of newsroom intake before that.

What it is

An intake layer for internal communications requests: a Google Form feeds a bound Sheet, the agent runs its three prompts, and the output lands in a triage log with a classification label, a revision diff, an escalation decision, and a confidence score per submission. No human in the loop unless the confidence threshold isn't met.

Single-shot LLM triage
One prompt tries to classify, rewrite, and route in a single pass. Every request pulls the full context window, and there is no principled place for the model to say "I'm not sure."
comms-triage-agent
Three prompts, each with a defined scope and a conditional exit. The knowledge-base load is gated per step, so a routine request stays cheap. Uncertainty has an explicit path: it goes to a person, with the confidence score attached.
What it does

Triage, revise, escalate. An incoming request hits the form and the agent classifies its intent and urgency, then either rewrites it against the org's comms standards or flags it for structural issues, then routes it to an owner or holds it for review. Each step is its own prompt with its own exit condition, so a request that resolves at step one never pays for steps two and three.

The knowledge base loads conditionally. Audience profiles, triage criteria, brevity rules, and org background live as separate knowledge-base documents, and each prompt pulls only the documents its classification says it needs. Not every request deserves the full context window; the architecture enforces that instead of hoping.

The numbers are design targets, and they say so. The ~160 hours-per-year figure is a projection: average pre-automation triage time per request, times monthly volume, times twelve, with an assumed human-review rate. It is labeled as a projection everywhere it appears, the constants are exposed in config.gs so another org can recalculate against its own baseline, and the triage log exists precisely so the projection can be validated against real throughput over time.

Domain depth over generic classification. The triage logic reflects how comms intake actually behaves in a large engineering org: executive requests need a high-touch template, partnership asks need a different one, low-touch email drafts need a rewrite and nothing else. The routing carries an accountability tier built for senior ICs. A wrong answer is expensive. A slow one is merely annoying.

How it works
form submission (bound Google Sheet) │ ▼ [1] triage prompt → classify intent + urgency (conditional KB load) │ ▼ [2] revision prompt → rewrite, or flag structural issues │ ▼ [3] escalation prompt → route to owner, or hold │ ├─▶ confidence ≥ threshold → Triage_Log + optional email └─▶ confidence < threshold → human review
01
Classify at intake
The triage prompt reads the submission, classifies intent and urgency, and decides which knowledge-base documents the rest of the run needs. Classification is the cheap step, so it runs first and everything downstream is scoped by it.
02
Revise or flag
The revision prompt rewrites the request against the org's comms standards, or flags it when the problem is structural rather than editorial. The output is a diff, not a replacement, so the requester can see exactly what changed and why.
03
Escalate or hold
The escalation prompt routes the request to its owner with a briefing, or holds it. High-touch executive and partnership requests get templated briefings; low-touch drafts go out with the revision attached.
04
Log everything
Every submission lands in the triage log: classification label, revision diff, escalation decision, confidence score. The log is not bookkeeping, it is the mechanism that lets the design targets be checked against reality.
05
Recalculate for your org
The volume and time-per-request constants are exposed in config.gs. A different org plugs in its own baseline and gets its own projection, instead of inheriting a number that was true somewhere else.
Why it matters

The interesting part of this system is not the automation, it is where the automation stops. A confidence threshold with a human on the other side of it is a design decision about trust: the agent handles the volume, and the moment it stops being sure, a person decides. And the metrics discipline is the same idea applied to reporting: a projection labeled as a projection, with its inputs exposed, is worth more to a serious reader than an impressive number with no math behind it.

The projection is labeled a projection, and the log exists so the number can be checked against reality. That honesty is a feature of the system, not a disclaimer on it.
