I'm Mitchell Williams, and relocation-os is a personal agentic operating system I built for relocating from Seattle to Spain, a decision with hundreds of interlocking questions. It turns that into an engineered system: a council of nine research agents, a locked-decision registry that keeps settled questions settled, and a mission pattern that makes every trip and evaluation a reproducible pipeline. The signature piece is the privacy quarantine. A fictional synthetic twin, same schema, invented facts, stands in for my private corpus, so the architecture stays demonstrable without publishing my actual life. A mechanical gate scans the exact bytes going public and proves no personal fact got through. It runs on a dual runtime, Claude Code and Codex, sharing one contract. What it does

Determinism first. Orchestration lives in scripts and structured pipelines, not in model improvisation. The mission pattern converts each scouting trip or city evaluation into a folder with a defined pipeline and auditable outputs, and a CI eval harness enforces invariant gates on every pull request. The model does the research; the system decides the shape.

Verified, not vibes. Research questions route through a council of nine agents and a multi-model research pipeline, then an adjudication pass that classifies every claim before it enters the knowledge base. A wrong fact about visa timelines or tax residency is expensive, so nothing lands in the KB on one model's say-so.

Locked decisions are locked. Settled constraints live in a registry the harness must consult before acting. No agent, no future session, re-litigates them. This is the anti-drift mechanism: the system's memory of why outlives any single context window.

The privacy quarantine. Personal data lives in gitignored private memory and never leaves. A fictional synthetic twin, same schema, invented facts, stands in for it so the architecture stays demonstrable. The public repo is generated by a sanitize script that copies tracked files only, then a marker scan runs against the exact bytes to be published. One hit and the publish stops. The public cut ships with fresh git history: zero archaeology.

How it works
a question (a trip, a city, a visa path) │ ▼ [1] mission → one folder, one pipeline, auditable outputs │ ▼ [2] council (6 mission roles + 3 standing roles) ├─▶ multi-model research (routed, cross-checked) └─▶ adjudication (every claim classified before it lands) │ ▼ [3] knowledge base → locked-decision registry (settled stays settled) │ ▼ [4] private memory (gitignored, never committed) ⇢ synthetic twin │ ▼ [5] privacy gate (marker scan on the exact published bytes) → public cut
01
One contract, two runtimes
The harness contract is written twice, once for Claude Code and once for Codex, and kept identical. Any agent session, on either runtime, inherits the same rules, the same memory discipline, and the same locked decisions.
02
Missions, not chats
Each trip or evaluation is a mission: a folder with a defined pipeline and outputs that can be audited later. Context is budgeted deliberately; tier-1 memory is an index measured in hundreds of tokens, and everything else loads lazily by topic.
03
Research through a council
Nine agents with defined roles run the research through multiple models, then an adjudication pass classifies every claim before it enters the knowledge base. Findings land with per-claim attribution, not as a wall of confident prose.
04
Decisions lock
Once a constraint is settled it enters the locked-decision registry, and the harness must consult the registry before acting. The system cannot be talked back into a question it already answered.
05
The quarantine holds, mechanically
The public repo is a generated artifact: a sanitize script copies tracked files only, a synthetic twin stands in for the private corpus, and a scan runs the full marker list against the exact bytes to be published. A single hit stops the publish.
Why it matters

Most agentic-system demos are toys because nothing is at stake. Here the stakes are a family's move across an ocean, which is exactly why the system is built the way production systems are built: deterministic where it can be, verified where it can't, and honest about the boundary between the two. The pattern generalizes to any personal-scale system that wants to show its architecture without publishing the life inside it.

The repo demonstrates the system. It does not publish the life. And that claim is not a promise, it is a mechanical gate that scans every byte before it ships.
